Overview

This lab targets ENCOR 2.2.b for IPsec tunneling. You will configure and verify a site-to-site IPsec VPN, then compare policy-based crypto maps with route-based tunnel protection so you can recognize both operational models.

Topology

INET-RTR is only transit. It must be able to route public transport addresses, but it must not learn private LAN prefixes.

Task 1: Crypto Baseline

Before configuring IPsec, prove that clear-text LAN traffic would fail or cross the wrong path and that peer transport addresses are reachable.

enable
terminal length 0
show ip route 198.51.100.6
show ip route 0.0.0.0
ping 198.51.100.6 source Ethernet0/0
show access-lists
show crypto ikev2 sa
show crypto ipsec sa

CML note: An exact lookup for the peer transport address may show % Subnet not in table because the peer is reached through the default route. Verify the default route and the sourced underlay ping.

Task 2: Configure Policy-Based IPsec

Use a crypto ACL to define interesting traffic between HQ and branch LANs. Configure both peers with reversed ACLs and matching IKEv2 parameters.

HQ-RTR

configure terminal
crypto ikev2 proposal ENCOR-PROP
 encryption aes-cbc-256
 integrity sha256
 group 14
!
crypto ikev2 policy ENCOR-POL
 proposal ENCOR-PROP
!
crypto ikev2 keyring ENCOR-KEYS
 peer BR-RTR
  address 198.51.100.6
  pre-shared-key local cisco123
  pre-shared-key remote cisco123
!
crypto ikev2 profile ENCOR-IKEV2
 match identity remote address 198.51.100.6 255.255.255.255
 authentication local pre-share
 authentication remote pre-share
 keyring local ENCOR-KEYS
!
ip access-list extended VPN-TRAFFIC
 permit ip 10.10.0.0 0.0.0.255 10.60.0.0 0.0.0.255
!
crypto ipsec transform-set ENCOR-TS esp-aes 256 esp-sha256-hmac
 mode tunnel
!
crypto map ENCOR-MAP 10 ipsec-isakmp
 set peer 198.51.100.6
 set transform-set ENCOR-TS
 set ikev2-profile ENCOR-IKEV2
 match address VPN-TRAFFIC
!
interface Ethernet0/0
 crypto map ENCOR-MAP
end

Interpretation: IOS XE warns while an IKEv2 proposal, policy, or profile is incomplete. Those warnings clear once encryption, integrity, DH group, proposal, authentication, identity match, and keyring are all configured.

Task 3: Compare Route-Based IPsec

On EDGE-RTR and DC-RTR, build virtual tunnel interfaces and apply an IPsec profile. This method makes the tunnel a routable interface and avoids crypto ACL route ambiguity.

EDGE-RTR

configure terminal
crypto ikev2 proposal ENCOR-PROP
 encryption aes-cbc-256
 integrity sha256
 group 14
!
crypto ikev2 policy ENCOR-POL
 proposal ENCOR-PROP
!
crypto ikev2 keyring ENCOR-KEYS
 peer DC-RTR
  address 203.0.113.6
  pre-shared-key local cisco123
  pre-shared-key remote cisco123
!
crypto ikev2 profile ENCOR-IKEV2
 match identity remote address 203.0.113.6 255.255.255.255
 authentication local pre-share
 authentication remote pre-share
 keyring local ENCOR-KEYS
!
crypto ipsec transform-set ENCOR-TS esp-aes 256 esp-sha256-hmac
 mode tunnel
!
crypto ipsec profile ENCOR-PROFILE
 set transform-set ENCOR-TS
 set ikev2-profile ENCOR-IKEV2
!
interface Tunnel20
 description Route-based IPsec VTI to DC-RTR
 ip address 172.16.20.1 255.255.255.252
 tunnel source Ethernet0/0
 tunnel destination 203.0.113.6
 tunnel mode ipsec ipv4
 tunnel protection ipsec profile ENCOR-PROFILE
 no shutdown
!
ip route 10.80.0.0 255.255.255.0 Tunnel20 172.16.20.2
end

Comparison: Crypto maps encrypt traffic selected by an ACL on an interface. VTIs encrypt traffic routed to a tunnel interface. Both are valid; verification must match the model.

CML note: Tunnel20 can show line protocol down before interesting traffic builds the IKE/IPsec SAs. Initiate a ping to the opposite tunnel address, then recheck the tunnel and crypto state.

Task 4: Verification Depth

Generate interesting traffic and verify both IKE and IPsec state. Do not rely on only the ping result.

Policy-Based IPsec

# HQ-RTR
ping 10.60.0.10 source 10.10.0.1 repeat 5
show crypto ikev2 sa detailed
show crypto ipsec sa peer 198.51.100.6
show crypto session detail
show access-lists VPN-TRAFFIC
!
# BR-RTR
show crypto ikev2 sa
show crypto ipsec sa peer 198.51.100.2
show crypto session detail
show access-lists VPN-TRAFFIC

Route-Based IPsec VTI

# EDGE-RTR
ping 172.16.20.2 source Tunnel20 repeat 5
ping 10.80.0.1 source 10.70.0.1 repeat 5
show interface Tunnel20
show crypto ikev2 sa
show crypto ipsec sa peer 203.0.113.6
show crypto session detail
show ip route 10.80.0.0
!
# DC-RTR
ping 172.16.20.1 source Tunnel20 repeat 5
ping 10.70.0.1 source 10.80.0.1 repeat 5
show interface Tunnel20
show crypto ikev2 sa
show crypto ipsec sa peer 203.0.113.2
show crypto session detail
show ip route 10.70.0.0

CML note: The validated VTI reachability tests use router tunnel addresses and router LAN-interface sources. If you extend testing to Alpine endpoint hosts, first prove the router-to-router VTI path, then troubleshoot endpoint default gateways separately.

Look for incrementing encaps and decaps counters. A one-way counter usually means peer ACL mismatch, NAT interference, wrong peer identity, or return routing failure.

Task 5: Troubleshooting

  1. IKEv2 fails because the remote identity does not match the profile.
  2. IKE is up but IPsec counters do not increment because the crypto ACL is reversed on one peer.
  3. Encaps increment but decaps remain zero because the return route points to the Internet instead of the VPN.
  4. A VTI remains down until traffic to the opposite tunnel address or routed prefix triggers IKE/IPsec negotiation.
  5. Interface-only static routes to a VTI are less explicit in verification output than routes that include the tunnel next-hop address.
debug crypto ikev2 protocol
debug crypto ikev2 error
show crypto ikev2 sa detailed
show crypto ipsec sa | include local ident|remote ident|pkts encaps|pkts decaps
show crypto session detail
show interface Tunnel20
show ip route 10.60.0.0
show ip route 10.80.0.0
show ip cef 10.80.0.1 detail
