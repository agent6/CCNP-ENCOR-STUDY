Overview

This capstone lab combines ENCOR 2.2.a and 2.2.b. You will place the public transport link in a front-door VRF, build an encrypted GRE tunnel over that transport, and carry segmented CORP routes across the overlay.

The main skill is reasoning about three routing contexts at the same time: the global table, the transport VRF, and the overlay/customer VRF.

Topology

Pattern: The front-door VRF isolates the WAN underlay. The tunnel interface carries customer routes. IPsec encrypts the GRE packets between the public peer addresses.

Task 1: Build the Front-Door VRF

Move the Internet-facing interface into a TRANSPORT VRF. Reapply the IP address after the VRF assignment.

configure terminal
vrf definition TRANSPORT
 description Internet underlay for encrypted tunnels
 address-family ipv4
 exit-address-family
!
interface ethernet0/0
 description WAN underlay to INET-RTR
 vrf forwarding TRANSPORT
 ip address 198.51.100.2 255.255.255.252
 no shutdown
!
ip route vrf TRANSPORT 0.0.0.0 0.0.0.0 198.51.100.1
end
show ip route vrf TRANSPORT
ping vrf TRANSPORT 198.51.100.6 source 198.51.100.2

Task 2: Build GRE Over the Transport VRF

Bind the tunnel source lookup to the front-door VRF while keeping the tunnel interface itself available for overlay routing.

configure terminal
interface tunnel10
 description GRE overlay over TRANSPORT VRF
 ip address 172.16.10.1 255.255.255.252
 tunnel source ethernet0/0
 tunnel destination 198.51.100.6
 tunnel vrf TRANSPORT
 tunnel mode gre ip
 no shutdown
end
show interface tunnel10
show ip route vrf TRANSPORT 198.51.100.6

If the platform uses a different syntax, look for the same behavior: the tunnel destination must be resolved in the transport VRF, not the global routing table.

Task 3: Protect the GRE Tunnel with IPsec

Apply an IPsec profile to Tunnel10. The peer address is still reached through TRANSPORT.

crypto ikev2 keyring ENCOR-KEYS
 peer BR-RTR
  address 198.51.100.6
  pre-shared-key local cisco123
  pre-shared-key remote cisco123
!
crypto ikev2 profile ENCOR-IKEV2
 match identity remote address 198.51.100.6 255.255.255.255
 authentication local pre-share
 authentication remote pre-share
 keyring local ENCOR-KEYS
!
crypto ipsec transform-set ENCOR-TS esp-aes 256 esp-sha256-hmac
 mode transport
!
crypto ipsec profile GRE-PROTECT
 set transform-set ENCOR-TS
 set ikev2-profile ENCOR-IKEV2
!
interface tunnel10
 tunnel protection ipsec profile GRE-PROTECT

Task 4: Carry CORP Routes Across the Overlay

Place the customer-facing LAN in VRF CORP and route it through the encrypted GRE tunnel. Do not leak TRANSPORT routes into CORP.

vrf definition CORP
 address-family ipv4
 exit-address-family
!
interface ethernet0/1
 description CORP LAN
 vrf forwarding CORP
 ip address 10.10.0.1 255.255.255.0
 no shutdown
!
interface tunnel10
 vrf forwarding CORP
 ip address 172.16.10.1 255.255.255.252
!
ip route vrf CORP 10.60.0.0 255.255.255.0 tunnel10
end
show ip route vrf CORP
show ip route vrf TRANSPORT
ping vrf CORP 10.60.0.10 source 10.10.0.1

Ordering trap: Applying vrf forwarding CORP to Tunnel10 removes its IP address. Reapply the tunnel IP after assigning the overlay VRF.

Task 5: Failure Analysis

Investigate these failures and identify which routing context is broken:

  1. Transport ping fails because the default route was added to the global table instead of TRANSPORT.
  2. Tunnel10 is down because the tunnel destination is resolved in the wrong VRF.
  3. IKEv2 is up but CORP traffic fails because Tunnel10 was protected but not placed in the CORP VRF.
  4. IPsec counters increment one way only because the branch peer uses a different source address.
show ip route vrf TRANSPORT 198.51.100.6
show ip cef vrf TRANSPORT 198.51.100.6 detail
show interface tunnel10
show ip route vrf CORP 10.60.0.0
show crypto ikev2 sa detailed
show crypto ipsec sa peer 198.51.100.6
