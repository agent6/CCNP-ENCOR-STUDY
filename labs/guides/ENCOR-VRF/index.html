Overview

This lab targets ENCOR 2.2.a: configure and verify VRF as a data path virtualization technology. You will create VRF-lite routing tables, move interfaces into the correct VRFs, build tagged VRF transit across one physical link, and validate control-plane separation.

The lab is CCNP-level because the success criteria are not just "ping works." You must prove which routing table owns each interface, explain why pings fail without a VRF context, and keep CORP and GUEST separated while allowing CORP-only access to a service prefix.

Topology

Use HQ-RTR as the campus edge and DC-RTR as the service edge. CORP and GUEST must remain separated end to end. The single router-to-router physical link is split into 802.1Q subinterfaces: VLAN 10 carries CORP and VLAN 20 carries GUEST.

Task 1: Baseline Audit

Start by identifying which interfaces are currently in the global routing table and which connected routes disappear when a VRF assignment is made.

show vrf
show ip interface brief
show ip route connected
show running-config interface ethernet0/1
show running-config interface ethernet0/2

Operational trap: On IOS XE, adding vrf forwarding or ip vrf forwarding to an interface removes the existing IP address from that interface. Reapply the address after the VRF is attached.

Task 2: Build VRFs

Create VRFs using the modern vrf definition syntax. Know the legacy ip vrf form, but use one style consistently in production.

! HQ-RTR
configure terminal
vrf definition CORP
 description Employee routing table
 address-family ipv4
 exit-address-family
!
vrf definition GUEST
 description Guest routing table
 address-family ipv4
 exit-address-family
!
interface ethernet0/0
 no ip address
 no shutdown
!
interface ethernet0/0.10
 encapsulation dot1Q 10
 vrf forwarding CORP
 ip address 172.16.10.1 255.255.255.252
!
interface ethernet0/0.20
 encapsulation dot1Q 20
 vrf forwarding GUEST
 ip address 172.16.20.1 255.255.255.252
!
interface ethernet0/1
 description CORP LAN
 vrf forwarding CORP
 ip address 10.10.0.1 255.255.255.0
 no shutdown
!
interface ethernet0/2
 description GUEST LAN
 vrf forwarding GUEST
 ip address 10.20.0.1 255.255.255.0
 no shutdown
!
no ip route 10.30.0.0 255.255.255.0 172.16.0.2
no ip route 10.40.0.0 255.255.255.0 172.16.0.2
no ip route 10.100.0.10 255.255.255.255 172.16.0.2
ip route vrf CORP 10.30.0.0 255.255.255.0 172.16.10.2
ip route vrf GUEST 10.40.0.0 255.255.255.0 172.16.20.2
end
!

configure terminal

end

Alternative syntax you may encounter: ip vrf CORP with ip vrf forwarding CORP. Verification expectations are the same, but the running configuration format differs.

Task 3: Prove Isolation

Verify the global table cannot see VRF routes, then verify each VRF can reach only its own connected and static routes.

show vrf
show ip route
show ip route vrf CORP connected
show ip route vrf GUEST connected
show ip route vrf CORP 10.30.0.0
show ip route vrf GUEST 10.40.0.0
ping vrf CORP 172.16.10.2 source 172.16.10.1
ping vrf GUEST 172.16.20.2 source 172.16.20.1
ping vrf CORP 10.30.0.10 source 10.10.0.1
ping vrf GUEST 10.40.0.10 source 10.20.0.1
traceroute vrf CORP 10.30.0.10 source 10.10.0.1

A normal ping 10.30.0.10 should fail until the correct VRF context is supplied.

Task 4: CORP-Only Service Reachability

Make the shared service loopback reachable from CORP only. Do not add a GUEST route to the service prefix, and do not solve the problem with a global default route.

! DC-RTR
configure terminal
interface loopback0
 description Shared service loopback - CORP only
 vrf forwarding CORP
 ip address 10.100.0.10 255.255.255.255
 no shutdown
end
show ip route vrf CORP 10.100.0.10
!
! HQ-RTR
configure terminal
ip route vrf CORP 10.100.0.10 255.255.255.255 172.16.10.2
end
show ip route vrf CORP 10.100.0.10
show ip route vrf GUEST 10.100.0.10
ping vrf CORP 10.100.0.10 source 10.10.0.1
ping vrf GUEST 10.100.0.10 source 10.20.0.1

The CORP ping should succeed and the GUEST lookup should report that the subnet is not in the table. Larger environments may use MP-BGP route targets for inter-VRF leaking, but this lab keeps the focus on ENCOR VRF-lite forwarding behavior.

Task 5: Troubleshooting

Fix these defects without changing the segmentation policy:

  1. HQ-RTR e0/2 has an IP address but is not in VRF GUEST.
  2. The remote CORP or GUEST route is missing because the tagged transit subinterface is not up in the same VRF on both routers.
  3. CORP can reach the shared service, but GUEST can also reach it because of an overbroad GUEST static route.
  4. A technician tests with ping 10.30.0.10 and reports failure even though CORP routing is correct.
show running-config | section vrf
show vrf
show ip interface brief
show running-config interface ethernet0/0.10
show running-config interface ethernet0/0.20
show ip route vrf GUEST 10.100.0.10
show ip cef vrf CORP 10.30.0.10
ping vrf CORP 10.30.0.10 source 10.10.0.1

CML note: This IOL-XE image does not support show ip interface brief vrf all. Use show vrf, normal show ip interface brief, and per-VRF routing-table commands instead.

Design note: A single global transit interface does not carry traffic into the remote VRF. Use tagged subinterfaces, separate physical links, tunnels, or another VRF-aware transport so packets arrive in the intended routing table on both routers.
