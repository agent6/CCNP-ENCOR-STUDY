Overview

This lab targets ENCOR 2.2.a: configure and verify VRF as a data path virtualization technology. You will create VRF-lite routing tables, move interfaces into the correct VRFs, validate control-plane separation, and selectively leak routes without collapsing segmentation.

The lab is CCNP-level because the success criteria are not just "ping works." You must prove which routing table owns each interface, explain why pings fail without a VRF context, and repair route leaking without adding broad default routes.

Topology

Use HQ-RTR as the campus edge and DC-RTR as the service edge. CORP and GUEST must remain separated end to end. A shared services loopback is reachable only after controlled route leaking is configured.

Task 1: Baseline Audit

Start by identifying which interfaces are currently in the global routing table and which connected routes disappear when a VRF assignment is made.

show vrf
show ip interface brief
show ip route connected
show running-config interface ethernet0/1
show running-config interface ethernet0/2

Operational trap: On IOS XE, adding vrf forwarding or ip vrf forwarding to an interface removes the existing IP address from that interface. Reapply the address after the VRF is attached.

Task 2: Build VRFs

Create VRFs using the modern vrf definition syntax. Know the legacy ip vrf form, but use one style consistently in production.

configure terminal
vrf definition CORP
 description Employee routing table
 address-family ipv4
 exit-address-family
!
vrf definition GUEST
 description Guest routing table
 address-family ipv4
 exit-address-family
!
interface ethernet0/1
 description CORP LAN
 vrf forwarding CORP
 ip address 10.10.0.1 255.255.255.0
 no shutdown
!
interface ethernet0/2
 description GUEST LAN
 vrf forwarding GUEST
 ip address 10.20.0.1 255.255.255.0
 no shutdown
end

Alternative syntax you may encounter: ip vrf CORP with ip vrf forwarding CORP. Verification expectations are the same, but the running configuration format differs.

Task 3: Prove Isolation

Verify the global table cannot see VRF routes, then verify each VRF can reach only its own connected and static routes.

show ip route
show ip route vrf CORP
show ip route vrf GUEST
ping vrf CORP 10.30.0.10 source 10.10.0.1
ping vrf GUEST 10.40.0.10 source 10.20.0.1
traceroute vrf CORP 10.30.0.10 source 10.10.0.1

A normal ping 10.30.0.10 should fail until the correct VRF context is supplied.

Task 4: Controlled Route Leaking

Leak only the shared service /32 from SERVICES into CORP. Do not leak GUEST into CORP, and do not solve the problem with a global default route.

configure terminal
ip route vrf SERVICES 10.100.0.10 255.255.255.255 loopback100
ip route vrf CORP 10.100.0.10 255.255.255.255 10.30.0.1 global
end
show ip route vrf CORP 10.100.0.10
show ip cef vrf CORP 10.100.0.10 detail

Static leaking is intentionally used here because it is transparent and easy to audit. Larger environments often use MP-BGP route targets, but that belongs to broader VPN design rather than this focused ENCOR VRF-lite task.

Task 5: Troubleshooting

Fix these defects without changing the segmentation policy:

  1. HQ-RTR e0/2 has an IP address but is not in VRF GUEST.
  2. CORP can reach the shared service, but GUEST can also reach it because of an overbroad static route.
  3. A technician tests with ping 10.30.0.10 and reports failure even though CORP routing is correct.
show running-config | section vrf
show ip interface brief vrf all
show ip route vrf GUEST 10.100.0.10
show ip cef vrf CORP 10.30.0.10
ping vrf CORP 10.30.0.10 source 10.10.0.1
