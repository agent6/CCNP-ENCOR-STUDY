Overview

Topology

Task 1: Interpret Broadcast Network Behavior

R1-HQ and R2-CORE share the IOL-L2 AREA0-SW segment. Ethernet0/0 must remain broadcast for both OSPFv2 and OSPFv3 so DR/BDR election is meaningful.

show ip ospf interface ethernet0/0
show ip ospf neighbor
show ip ospf database network
show ipv6 ospf interface ethernet0/0
show ipv6 ospf neighbor

If R2-CORE shows Ethernet0/0 as point-to-point, repair only that interface. A mixed broadcast/point-to-point adjacency can still reach FULL, but the lab goal is to interpret the correct network type.

R2-CORE# configure terminal
interface Ethernet0/0
 no ip ospf network point-to-point
 no ipv6 ospf network point-to-point
end

On broadcast networks, FULL adjacency is expected with the DR/BDR. 2-WAY can be normal between DROTHER routers on a larger segment.

Task 2: Convert Routed Links to Point-to-Point

On true router-to-router links, use point-to-point network type to avoid unnecessary DR election. Also enable the direct R1-HQ to R2-CORE Area 0 link for both OSPFv2 and OSPFv3.

R1-HQ# configure terminal
router ospf 10
 no passive-interface Ethernet0/2
 network 10.0.12.0 0.0.0.3 area 0
exit
interface Ethernet0/1
 ip ospf network point-to-point
 ipv6 ospf 10 area 10
 ipv6 ospf network point-to-point
exit
interface Ethernet0/2
 ip ospf network point-to-point
 ipv6 ospf 10 area 0
 ipv6 ospf network point-to-point
exit
ipv6 router ospf 10
 no passive-interface Ethernet0/1
 no passive-interface Ethernet0/2
end
R2-CORE# configure terminal
router ospf 10
 no passive-interface Ethernet0/2
 network 10.0.12.0 0.0.0.3 area 0
exit
interface Ethernet0/1
 ip ospf network point-to-point
 ipv6 ospf 10 area 20
 ipv6 ospf network point-to-point
exit
interface Ethernet0/2
 ip ospf network point-to-point
 ipv6 ospf 10 area 0
 ipv6 ospf network point-to-point
exit
ipv6 router ospf 10
 no passive-interface Ethernet0/1
 no passive-interface Ethernet0/2
end
show ip ospf interface brief
show ipv6 ospf interface brief
show ip ospf neighbor detail

Alternative: on some platforms and OSPFv3 address-family style, use ospfv3 network point-to-point or the equivalent interface command shown by the parser.

Task 3: Passive Interfaces with Intent

Use passive-interface default, then disable passive mode only on real neighbor links. LAN prefixes should still advertise, but they should not send Hellos toward hosts.

 ip ospf network point-to-point
 ipv6 ospf 10 area 10
 ipv6 ospf network point-to-point

Task 4: Mismatch Defect Analysis

Repair these without replacing the whole OSPF configuration:

  1. R1-HQ uses broadcast while R2-CORE uses point-to-point on the shared segment.
  2. R3-DIST has an OSPF hello/dead timer mismatch on its point-to-point uplink.
  3. R4-BR cannot form a neighbor because the transit interface is still passive.
  4. OSPFv3 is configured, but the interface-level OSPFv3 network type does not match the neighbor.

Use focused show commands before changing anything. The repair should explain which symptom proved each defect.

show ip ospf interface
show ip ospf neighbor detail
show logging | include OSPF
debug ip ospf hello
show ipv6 protocols
show ipv6 ospf interface brief

Task 5: Verification Across OSPFv2 and OSPFv3

show ip ospf neighbor
show ipv6 ospf neighbor
show ip ospf interface brief
show ipv6 ospf interface brief
show ip route ospf
show ipv6 route ospf
ping 10.20.20.10 source 10.0.13.1
ping 10.30.30.10 source 10.0.13.1
ping 2001:db8:20:20::10 source 2001:db8:0:13::1
ping 2001:db8:30:30::10 source 2001:db8:0:13::1

The source address must exist on the device where you run the ping. R1-HQ does not own the HQ user LAN address 10.10.10.1; that address is on R3-DIST.

Troubleshooting Notes

  • If a router displays a non-blocking startup warning for no ip domain-lookup, continue the lab. The packaged YAML has been cleaned so this warning should not appear on a fresh import.
  • A mixed OSPF network type can still show a FULL neighbor. Always verify the interface type with show ip ospf interface or show ipv6 ospf interface, not neighbor state alone.
  • Timer mismatches usually leave the interface selected for OSPF but prevent a stable neighbor. In this lab, R3-DIST Ethernet0/0 should return to default hello/dead timers to match R1-HQ.
  • A passive transit interface advertises its connected subnet but sends no Hellos. If an expected neighbor is missing, compare show ip protocols passive interfaces with the physical topology.
  • OSPFv3 is enabled per interface in this lab. IPv4 OSPF network statements do not enable OSPFv3.
  • The R5-SVC router icon sits directly above APP-SERVER in the CML topology. If the console says app-server login:, close that console and right-click the router icon above it.
  • If the browser console drops the first characters of a command and IOS reports an invalid input marker, retype the exact command at the prompt before troubleshooting the network.
  • One dropped packet on the first ping can be ARP or IPv6 neighbor discovery. Use the final success rate after the adjacency and route tables are stable.
