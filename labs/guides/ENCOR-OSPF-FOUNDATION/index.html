Overview

This lab targets ENCOR 3.2.b by configuring simple OSPFv2 and OSPFv3 environments and interpreting neighbor state, router IDs, passive interfaces, and route installation. You will use both network-statement and interface-level styles so you can recognize either in the field.

Task 2: Configure OSPFv2

Use classic OSPFv2 network statements and explicitly set point-to-point network type on routed transit links. Leave user and server LANs passive so their prefixes advertise without sending Hellos toward endpoints.

R1-HQ

router ospf 10
 router-id 1.1.1.1
 passive-interface default
 no passive-interface Ethernet0/0
 no passive-interface Ethernet0/1
 no passive-interface Ethernet0/2
 network 1.1.1.1 0.0.0.0 area 0
 network 10.0.0.0 0.0.0.255 area 0
 network 10.0.12.0 0.0.0.3 area 0
 network 10.0.13.0 0.0.0.3 area 10
!
interface Ethernet0/1
 ip ospf network point-to-point
!
interface Ethernet0/2
 ip ospf network point-to-point

You may also see newer OSPFv3 address-family syntax in production, such as router ospfv3 10 with address-family ipv6 unicast and interface command ospfv3 10 ipv6 area 0. Be able to recognize it, but use the classic syntax above for this CML lab.

Task 4: Passive Interfaces

Confirm the passive design. Transit links must be non-passive. LAN interfaces to HQ-USER, BR-USER, and APP-SERVER should advertise their connected prefixes but never form OSPF neighbors.

show ip ospf interface brief
show ipv6 ospf interface brief
show ip protocols
show ipv6 protocols

On point-to-point transit links, expect FULL/-. On the Area 0 Ethernet segment through AREA0-SW, expect DR/BDR neighbor roles. A passive OSPF interface still advertises its connected prefix; it simply does not send Hellos or form adjacencies on that interface.

Task 5: Verification

Verify adjacencies, route installation, and data-plane reachability. Start with neighbor state, then prove routes and pings. Do not rely on ping alone.

show ip ospf neighbor detail
show ipv6 ospf neighbor detail
show ip route ospf
show ipv6 route ospf
show ip ospf database router
show ipv6 ospf database router

Useful reachability checks from R1-HQ:

ping 10.30.30.10 source 10.0.13.1
ping 10.20.20.10 source 10.0.12.1
ping 2001:db8:30:30::10 source 2001:db8:0:13::1
ping 2001:db8:20:20::10 source 2001:db8:0:12::1

Useful reachability checks from R5-SVC:

ping 10.10.10.10
ping 10.20.20.10
ping 2001:db8:10:10::10
ping 2001:db8:20:20::10

Troubleshooting Notes from CML Validation

  • The IOL-XE image used for this lab rejects no ip domain-lookup during startup. It is not required for the lab and has been removed from the YAML.
  • If a routed transit adjacency does not look right, verify both sides use ip ospf network point-to-point and ipv6 ospf network point-to-point on the /30 and IPv6 transit links.
  • FULL/- is expected on point-to-point links. DR and BDR roles are expected on the shared Area 0 Ethernet segment through AREA0-SW.
  • If a LAN prefix is missing from the route table, check whether the interface was selected by the OSPFv2 network statement or OSPFv3 interface command. A passive interface can still advertise the prefix.
  • If IPv6 OSPF neighbors are missing while IPv4 OSPF works, check for the interface-level ipv6 ospf 10 area X command. OSPFv3 is not enabled by IPv4 network statements.
  • Console input can occasionally drop the first characters of a command in the browser terminal. If a command returns an unexpected parser error but the syntax is valid, retype that command at the prompt.
  • The first ping may fail while ARP or IPv6 neighbor discovery resolves. Confirm with the full success rate and routing table before treating one dropped packet as an OSPF issue.
