Overview

This lab targets ENCOR 2.2.b for GRE tunneling. You will build an overlay between HQ-RTR and BR-RTR across INET-RTR, route branch prefixes over the tunnel, and verify the difference between underlay reachability, tunnel line protocol, and routed overlay reachability.

Topology

GRE rides over the transport network. The tunnel source and destination must be reachable in the underlay before any overlay routing can work.

Task 1: Validate the Underlay

Confirm HQ-RTR and BR-RTR can reach each other across INET-RTR using only transport addresses. Do not configure a tunnel until the underlay is stable.

show ip route 0.0.0.0
show ip route 198.51.100.0
show ip cef 198.51.100.6
ping 198.51.100.6 source ethernet0/0
traceroute 198.51.100.6 source ethernet0/0
show cdp neighbors detail

CML note: The remote tunnel endpoint may be reachable through the default route, so an exact show ip route 198.51.100.6 lookup can report that the subnet is not in the table while the underlay ping still succeeds.

Task 2: Configure GRE

Build Tunnel10. Use physical interface sources first, then evaluate the loopback-source alternative.

! HQ-RTR
configure terminal
interface tunnel10
 description GRE overlay to BR-RTR
 ip address 172.16.10.1 255.255.255.252
 tunnel source ethernet0/0
 tunnel destination 198.51.100.6
 tunnel mode gre ip
 no shutdown
end
show interface tunnel10 | include line protocol|Tunnel source
!
! BR-RTR
configure terminal
interface tunnel10
 description GRE overlay to HQ-RTR
 ip address 172.16.10.2 255.255.255.252
 tunnel source ethernet0/0
 tunnel destination 198.51.100.2
 tunnel mode gre ip
 no shutdown
end
show interface tunnel10 | include line protocol|Tunnel source

Design depth: A physical tunnel source is simple but fails if that interface goes down. A loopback tunnel source survives underlay path changes, but only if the loopback is advertised or statically routed in the underlay.

Task 3: Route Through the Overlay

Use static routes first, then compare with a lightweight routing protocol over the tunnel.

! HQ-RTR
configure terminal
ip route 10.60.0.0 255.255.255.0 tunnel10 172.16.10.2
end
show ip route 10.60.0.0
show ip cef 10.60.0.10 detail
ping 10.60.0.10 source 10.10.0.1
traceroute 10.60.0.10 source 10.10.0.1
! BR-RTR
configure terminal
ip route 10.10.0.0 255.255.255.0 tunnel10 172.16.10.1
end
show ip route 10.10.0.0
ping 10.10.0.10 source 10.60.0.1

Static routing makes the data path obvious. OSPF or EIGRP over GRE can remove manual route maintenance, but you must still verify neighbor formation, passive-interface choices, and tunnel MTU.

Task 4: MTU, MSS, and Keepalive

GRE adds overhead. Validate the largest safe payload and configure TCP MSS adjustment if user traffic crosses the tunnel.

! Apply on HQ-RTR and BR-RTR.
configure terminal
interface tunnel10
 ip mtu 1476
 ip tcp adjust-mss 1436
 keepalive 10 3
end
show interface tunnel10 | include MTU|line protocol|Keepalive|Tunnel source
!
! HQ-RTR
ping 10.60.0.10 size 1476 df-bit source 10.10.0.1
ping 10.60.0.10 size 1477 df-bit source 10.10.0.1

With tunnel MTU set to 1476, the 1476-byte DF-bit ping should succeed and the 1477-byte DF-bit ping should fail. That gives a concrete MTU boundary.

Task 5: Troubleshooting

Fix these defects while keeping the overlay design intact:

  1. Tunnel10 is up/down because the tunnel destination is not reachable.
  2. Tunnel10 is up/up but branch LAN traffic exits the Internet underlay due to a more specific global route.
  3. Large pings fail with DF set because tunnel MTU was not considered.
show interface tunnel10
show ip route 198.51.100.6
show ip route 10.60.0.10
show ip cef exact-route 10.10.0.10 10.60.0.10
debug tunnel keepalive

CML note: show tunnel interface tunnel10 is not supported on this IOL-XE image. Use show interface tunnel10, show running-config interface tunnel10, route output, and CEF output instead.
